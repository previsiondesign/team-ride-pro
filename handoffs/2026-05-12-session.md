## Why two layers of guard (not just one)
- **SQL partial unique index** `idx_rides_date_unique_active` on `rides(date) WHERE active`. Catches *all* paths (frontend, edge function, manual SQL, future code) and race conditions. Mirrors `getNextRide()`'s active-row filter exactly.
- **JS DB pre-check** in `ensureRidesFromSchedule()`. Avoids the constraint-violation error in the common case (better UX, no console errors). Catches unique-violation (`23505` / `idx_rides_date_unique_active`) and refetches the existing row instead of pushing a phantom local ride.

`cleanup_duplicate_rides` edge action keeps oldest, soft-deletes the rest — would have produced the same outcome as the manual cleanup. Worth knowing for future incidents.

## Attendance poll workflow — misdiagnosis & real fix
User saw "All jobs have failed" email and assumed cron was firing post-season. Logs proved otherwise:
```
post_poll_if_due → HTTP 200 {"skipped": true, "reason": "No upcoming practice found"}   ✅
send_reminders   → HTTP 504 after 2m40s hang   ❌ (Supabase gateway timeout)
```
Both actions already early-return when `getNextRide()` is null. The failure was a transient Supabase blip, would've failed mid-season too. But the user's underlying ask — "stop the off-season noise" — is still legit. Fix: workflow now calls `post_poll_if_due` first; if its body contains `"No upcoming practice found"`, skips `send_reminders` and exits green. Schedule stays defined → auto-resumes next season when practices exist (no `gh workflow disable` / re-enable footgun). Plus `call_edge()` helper retries 000/502/503/504 up to 3× (15s, 30s backoff) with `curl --connect-timeout 15 --max-time 90`. Manual `workflow_dispatch` runs bypass season gating so test_poll etc. work off-season.

User chose **graceful skip** (over hard-disable) and **retry on transient 5xx** via AskUserQuestion.
